# 文档汉化设计

## 背景

本仓库以章节形式记录 `acwj`（A Compiler Writing Journey）的编译器实现过程，
主要内容分布在顶层 [Readme.md](/home/shenli/acwj/.worktrees/feat-docs-zh-cn/Readme.md)、
`00_Introduction` 到 `64_6809_Target` 下的各章 `Readme.md`，以及
`64_6809_Target/docs/NOTES.md`。

用户希望先完成文档汉化，不处理代码内的字符串、注释和标识符，并要求在独立分支中完成工作。

## 目标

- 将仓库中的主要 Markdown 文档汉化为中文。
- 保持代码块、命令、路径、变量名、函数名和链接目标不变。
- 对常用术语采用“中文（英文）”首次出现说明，后续保持一致。
- 新增一个仓库级 `AGENTS.md`，为后续协作者提供工作约束。

## 范围

本次纳入范围的文件：

- 顶层 `Readme.md`
- `00_Introduction/Readme.md` 到 `64_6809_Target/Readme.md`
- `64_6809_Target/docs/NOTES.md`
- 仓库级 `AGENTS.md`

本次不纳入范围：

- `.c`、`.h`、`.s` 等源码文件
- 代码块内部的英文注释和标识符
- 图片资源、PDF 资源和测试输入文件

## 翻译规范

- 标题、正文、列表、注释性图片说明全部汉化。
- 代码块、命令、路径、链接地址、变量名、函数名、结构体名、枚举名不翻译。
- 对 `C`、`C++`、`ARM`、`QBE`、`AST`、`token` 等常见术语可直接保留。
- 对“词法分析（lexical analysis）”“语义分析（semantic analysis）”等已有稳定译法的术语，
  在首次出现时保留英文参考。
- 优先保证中文可读性，同时避免偏离原文技术含义。

## 执行策略

1. 在独立分支 `feat-docs-zh-cn` 中进行全部文档改动。
2. 先补齐仓库级文档：设计文档、实施计划、`AGENTS.md`。
3. 先汉化顶层入口文档和早期章节，固定术语与语气。
4. 继续按章节推进其余文档，并在每批次后进行一致性检查。
5. 最后扫描残留英文、检查 Markdown 链接和工作区状态。

## 风险与处理

- 文档总量较大，术语漂移风险高。
  处理方式：先写规范，再按统一规则批量推进。
- 部分文档包含大量代码和图片引用。
  处理方式：只翻译说明文字，避免破坏代码和链接。
- `master` 工作区未提交 `.worktrees/` 忽略规则。
  处理方式：真实改动全部保留在特性分支与 worktree 中，避免把汉化工作直接压到主分支。
